# Frontend Design Spec

**Date:** 2026-04-29
**Status:** Approved

## Overview

Add a functional, minimalist web frontend to the travel agent LangGraph project. The frontend communicates with a new FastAPI backend that wraps the existing pipeline. The API also serves as integration point for a future AI agent (hermes-agent).

**Tech stack:** FastAPI (Python) + Svelte (TypeScript, Vite)

## Requirements

1. **Model selector** вЂ” dropdown populated from backend `AVAILABLE_MODELS`, with option for custom model name input
2. **Output window** вЂ” live streaming log of pipeline progress via SSE (terminal-style, auto-scroll)
3. **Custom output directory** вЂ” text input to override the base output path (default: `output/`). The timestamped subdirectory convention (`<base>/<timestamp>/`) is preserved.
4. **Drag-and-drop file upload** вЂ” accepts `.gpx`, `.jpg/.jpeg/.png`, `.txt` files; shows file list with remove buttons. Exactly one `.gpx` file is required to run the pipeline. Images and `.txt` notes file are optional.
5. **Notes input** вЂ” textarea for tour notes fed into the blog generation prompt (existing `load_tour_notes` node)
6. **Run button** вЂ” triggers the pipeline, shows status (idle / running / done / failed)

FastAPI stays in the same Python project (`app/api/`). The Svelte frontend lives in a `frontend/` subdirectory, built to static files served by FastAPI in production. Two separate dev servers in development (Vite on :5173, FastAPI on :8000, Vite proxies `/api`).

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/pipeline/run` | Start pipeline; multipart form with `gpx_file`, `image_files[]`, `notes_file`, JSON fields `model`, `output_dir`. Returns `{"run_id": "<uuid>"}` |
| `GET` | `/api/pipeline/stream/{run_id}` | SSE endpoint; live progress events from pipeline nodes |
| `GET` | `/api/pipeline/result/{run_id}` | Final result: markdown, HTML, output file paths |
| `GET` | `/api/models` | List available Ollama models |
| `POST` | `/api/files/upload` | Upload a file (chunked); returns `{"filename": "...", "path": "..."}` |
| `DELETE` | `/api/files/{filename}` | Remove a previously uploaded file |

### SSE Event Protocol

```
event: progress
data: {"stage": "<node_name>", "status": "running|done", "message": "<human text>"}

event: error
data: {"stage": "<failed_stage>", "message": "<error description>"}

event: done
data: {"run_id": "<uuid>", "status": "success|failed", "output_dir": "<path>"}
```

Each pipeline node wrapper emits a `running` event before the real node executes and a `done` event after. Service code itself is unchanged вЂ” the graph orchestration layer handles event emission.

### State Management

A Svelte writable store `pipeline.ts` holds:
- `logLines: string[]` вЂ” accumulated progress messages
- `runState: "idle" | "running" | "done" | "failed"`
- `currentRunId: string | null`
- `result: RunResult | null` вЂ” blog post data after completion

## File Management

Uploaded files land in `data/uploads/<session_id>/`. Session ID is a UUID generated by the frontend on page load, stored in `sessionStorage`, and sent as a cookie or query parameter with each request. On `POST /api/pipeline/run`, files are referenced by their stored paths. The pipeline uses its existing `data/images/` and `data/gpx/` conventions internally. No automatic cleanup вЂ” files persist until a new session starts.

## Error Handling

- **Frontend SSE disconnect:** reconnect with exponential backoff (max 3 retries), then show "Connection lost" banner
- **File upload failure:** red badge on file card with error tooltip
- **Pipeline failure:** SSE emits `error` event, OutputWindow shows red log line, RunButton returns to idle
- **Backend validation:** `422` with field-level errors for missing GPX, no images, invalid model
- **Unknown run IDs:** `404` on stream/result endpoints
- **Ollama connection error:** caught in blog generator node, forwarded as SSE error event

## Integration with Existing Code

- **Nodes** вЂ” no changes. The graph wraps each node call to emit SSE events.
- **Services** вЂ” no changes.
- **`app/state.py`** вЂ” no changes.
- **`app/graph.py`** вЂ” adds optional `event_emitter` callback parameter to `build_graph()`. When present, nodes emit progress events. When absent, pipeline runs as before (CLI mode preserved).
- **New files:** `app/api/__init__.py`, `app/api/routes.py`, `app/api/events.py`, `app/api/server.py`

## Dependencies Added

**Python (`pyproject.toml`):**
- `fastapi` вЂ” web framework
- `uvicorn[standard]` вЂ” ASGI server
- `sse-starlette` вЂ” SSE support
- `python-multipart` вЂ” file upload parsing

**Node (`frontend/package.json`):**
- `svelte` вЂ” UI framework
- `@sveltejs/vite-plugin-svelte` вЂ” Vite integration
- `vite` вЂ” build tool
- `typescript` вЂ” type safety
- `vitest` + `@testing-library/svelte` вЂ” testing

## Testing

| Layer | Tool | Scope |
|-------|------|-------|
| API routes | `FastAPI TestClient` + pytest | Mock pipeline, assert status codes, validation errors |
| SSE streaming | `httpx.AsyncClient` | Stream endpoint, verify event format, ordering, completion |
| Svelte components | Vitest + `@testing-library/svelte` | Render tests, store reactivity, EventSource mocking |
| Full pipeline | Manual | Start servers, upload files, run pipeline, verify output |

## Non-Goals

- User authentication / multi-user support
- Persistent run history / database
- Mobile-responsive layout (desktop-first)
- Dark mode toggle (single dark theme for terminal output)
- File type validation beyond accept filter (server validates)
